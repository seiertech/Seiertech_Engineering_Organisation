## 1. Source Lesson

Two compounding lessons. First: the 8-persona depth gap (Team 1 personas written via a batch script, missing 11 of 19 standard sections) was only caught because a founder explicitly requested a sense-check — nothing in the EMS structurally required that check to happen, and an equivalent gap could easily recur the next time multiple personas are written quickly. Second, and more fundamental: every lesson learned across this entire build session — the QA violation, the register gap, the three simulation bugs, this very pattern — existed only as prose in commit messages and one-off audit documents. `OPR-000008` Knowledge Capture had always instructed personas to "extract lessons learned," but no conformant register existed to receive them, and nothing required a lesson to actually change doctrine rather than just be observed and left.

## 3. Before / After

**Before:** No mechanism existed for a finding to become anything more durable than a commit message or a session-specific markdown file with no required structure, no required link to what it changed, and no requirement that it ever resolve.

**After:** `REG-000010` is the single place every lesson is recorded with a mandatory schema (Finding, Impact, Doctrine Amendment reference, Reuse Guidance). `OPR-000012` makes it structurally mandatory: a lesson at status CAPTURED with no amendment is a hard stop at the start of the next significant work session — not a suggestion, a gate.

---

## 4. Why This Amendment Resolves The Lesson

This is the amendment that prevents the other six lessons in this register from being the last of their kind. It does not promise that no future bug will occur — it promises that when one does, there is now a structural reason it must become reusable doctrine rather than a one-off fix that the next session has to rediscover independently. The Step 0 session-start gate in `OPR-000012` is the specific mechanism that closes the loop: it is the thing that would have caught, automatically, that this exact problem (lessons not compounding) had been forming across the whole session before it was ever explicitly named.

---

## 5. Verification

Verification here is necessarily different from the other two amendments — this is a process change, not a code or content fix with an immediate pass/fail check. Verification consists of: (a) the register and operation are themselves conformant to STD-000002/STD-000003, confirmed by manual read against the same structure used for REG-000008/009; (b) all 7 lessons from this session are backfilled with real, specific Findings (not vague placeholders) as a proof of the schema actually working in practice, not just in theory; (c) this very document is itself the first real exercise of the mechanism it describes — a Doctrine Amendment produced for a doctrine-mechanism lesson, applying the process to itself.

---

## 6. Regression Prevention

The regression prevention for this lesson IS the amendment itself — `OPR-000012` Step 0 is a standing, permanent gate, not a one-time check. The honest limitation: this gate currently relies on a persona or the founder remembering to run it at session start, since no script yet enforces it automatically. That is itself a candidate future lesson if it's ever skipped — worth watching for, not yet built as automation.

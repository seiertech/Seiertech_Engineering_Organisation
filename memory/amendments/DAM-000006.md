## 1. Source Lesson

LES-000011 — no mechanism existed to persist a platform's target repo URL beyond the moment of intake, despite later forward missions needing it to deliver anywhere. Found while building the cross-repo delivery script this amendment introduces.

This amendment directly addresses the two fundamentals the founder identified as highest-leverage after `DAM-000005`: the gap between "Release decision reached" and "anything actually happens in the target repo."

## 3. Before / After

**Before:** A BUILD mission could reach a real RELEASE decision and nothing happened in the target platform repository — the loop terminated at a decision, not an artefact. No script had write access to any repo outside the EMS repo itself.

**After:** A RELEASE decision triggers `deliver_to_target_repo.py`, which clones the target repo (using a separate, explicitly-required `TARGET_REPO_TOKEN`), creates a branch, commits the EDP to a fixed path (`.ems/missions/MSN-{issue}_EDP.md`), and opens a real GitHub Pull Request describing what was proposed.

---

## 4. Honest Scope — What This Amendment Does NOT Claim

This is recorded explicitly because overclaiming here would be a worse outcome than the gap itself. The Engineering Delivery Package produced by `run_build_chain.py` is markdown describing intended changes, not a diff or file contents — there is no patch for this script to apply. **No source file in any target repository is modified by this amendment.** What is delivered is a structured, committed proposal — visible, real, and actionable by a human or a future builder agent (Kiro), but not autonomous code generation. The PR body and the committed EDP file both state this plainly to anyone who encounters them, not just in this amendment record.

---

## 5. Why This Amendment Resolves The Lesson

LES-000011 is resolved by writing the URL down at the one point it's reliably known (intake), in a location any later script can read without re-deriving or re-requesting it. The cross-repo write capability fundamental is resolved to the extent it honestly can be at v1 — the system now reaches across the repo boundary and produces something real in the target platform, which it could not do before. The remaining gap (proposal → actually-applied code) is named precisely, not blurred, so the next increment has a clear, specific target rather than a vague "finish the loop" instruction.

## 6. Verification

- Built a real local fixture: a `git init`-ed local repo standing in for a target platform, plus a fixture EMS-side platform directory with a real EDP, TDA verdict, and `PLATFORM_REPO_URL.txt` pointing at the fixture via a `file://` URL
- Ran `deliver_to_target_repo.py` against this fixture for real — confirmed clone, branch creation (`mission/MSN-3002`), file write, commit, and push all worked correctly via real `git` operations, not mocked
- Manually inspected the resulting branch in the fixture repo: confirmed the committed file exists at the designed path with the correct content, including the "this is a proposal, not an applied change" framing baked into the artefact itself
- Confirmed the script degrades gracefully (clear message, no crash) at the GitHub-API-specific PR-creation step, which a `file://` fixture cannot satisfy — this is the honest limit of what could be verified without a real GitHub repo and token
- Confirmed clean failure without `TARGET_REPO_TOKEN` and without an existing EDP, both before any network/git call is attempted
- Full existing 10-test unit suite re-run, all pass; YAML re-validated after the workflow change

---

## 7. Regression Prevention

No automated test was added for the live git/GitHub-API mechanics specifically (same limitation noted in `DAM-000005` for GitHub Actions job wiring — not unit-testable with the current setup). The fixture-based manual verification performed here is the closest available substitute and is the same standard applied to the brownfield/greenfield simulation work earlier this session.

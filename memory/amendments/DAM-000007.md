## 3. Before / After — Per Finding

**LES-000012 (false citation claim):** Before — `EMS_OPERATING_MODEL.md` Section 8 claimed to cite "AUTH-001 through AUTH-010" and "REG-000001 through REG-000010" while actually referencing only 6 of each. After — every one of the 10 authorities and 10 registers is genuinely cited somewhere in the document body, woven in at points where each substantively belongs (e.g. `AUTH-009` Release Authority added to the release-decision row of the decision table, `REG-000007` Build Governance Register added at the intake/sync point in the lifecycle diagram), and Section 8's table now states this accurately rather than as a round-number claim.

**LES-000013 (stale persona references):** Before — neither Documentation and Knowledge Curator persona file mentioned `OPR-000012`, despite that operation existing specifically to govern the lesson-curation activity central to this persona's job. After — both reference it at the appropriate section.

**LES-000014 (triplicated dead code):** Before — `call_nim.py` existed but was imported by nothing; three chain scripts each carried an identical inline copy. After — all three import the single canonical function. A real, if minor, signature mismatch was caught and fixed in the process: the canonical file's `max_tokens` parameter had no default, while every call site relied on one.

---

## 4. Why This Amendment Resolves The Lesson

Each fix closes the gap at the point it actually exists, not adjacent to it: the operating model's citations were corrected by genuinely weaving the missing IDs into substantive content, not by padding the document with a list. The persona staleness was fixed by editing the persona files themselves, the only place a future instance of that persona would actually read its own instructions from. The triplicated code was resolved by consolidation, not by adding a comment noting the duplication — the risk LES-000014 named only goes away when there is structurally one copy left to drift.

## 6. Regression Prevention

The `call_nim` consolidation is itself the regression prevention for LES-000014 — there is structurally one copy to maintain going forward, so the next fix to NIM call logic cannot silently fail to propagate. For LES-000012 and LES-000013 (doctrine drift after new artefacts are added), no automated check exists yet; the general pattern — sweep periodically, check claims against reality programmatically rather than trusting them — is the only mitigation currently in place, consistent with the limitation already disclosed in `DAM-000005`/`DAM-000006` for GitHub Actions wiring and git mechanics, which are similarly not yet covered by the `ems_engine` automated test suite.

## 1. Source Lesson

LES-000008: no single authoritative document explains how autonomous engineering work actually flows through the EMS end to end — the model exists only as the sum of 12 separate Operations documents. LES-000009: the system's governance documentation has substantially outpaced its executed work — a rich, audited doctrine layer with no closed execution loop. Both surfaced by an external review (30 June 2026) and independently consistent with the prior session verdict on this EMS.

---

No existing Operations, Standards, Authorities, Registers, or Personas were modified, merged, or retired. Per the review's own immediate recommendations (preserve existing structure, no restructuring without justified architecture) and the founder's explicit direction this session, this amendment is additive and synthesising only.

---

## 3. Before / After

**Before:** The operating model existed implicitly, assembled by reading 12 separate Operations documents and inferring how they connect. No document answered, in one place: how work enters the system, how it progresses, how decisions are made, how authority is applied, how work completes.

**After:** `EMS_OPERATING_MODEL.md` exists as a single synthesising document answering exactly those five questions, built entirely from the existing 12 Operations (no new process invented) plus an honest statement of the imbalance LES-000009 identifies and a concrete resequencing recommendation.

---

## 4. Why This Amendment Resolves The Lesson

LES-000008 is resolved structurally: the document exists, is reachable from the doctrine inventory, and synthesises rather than duplicates the 12 Operations — so it stays accurate as those Operations evolve rather than becoming a second, divergent source of truth. LES-000009 is resolved by being stated plainly inside the same document, with a specific, sequenced recommendation (build the BUILD mission executor before further governance expansion) rather than left as an abstract concern — turning the review's finding into an actionable constraint on what gets built next.

---

## 5. Verification

Verification here is a coherence check, not a pass/fail test: does `EMS_OPERATING_MODEL.md` actually answer all five questions the review posed, using only artefacts that already exist (no invented process), and does it correctly cross-reference every Operation it synthesises. Confirmed by direct construction — each of the document's five sections is built by walking the relevant Operations in sequence and citing them by ID, not by independent narrative.

---

## 6. Regression Prevention

The risk this amendment is exposed to is document drift — the Operating Model going stale as Operations are added or changed. Mitigation: the document explicitly states it is a synthesis, not a primary source, and any future Operation addition or change should include a corresponding update to `EMS_OPERATING_MODEL.md` as part of that change, the same discipline already required for relationship-graph updates. This is recorded as guidance in the document itself rather than as a new automated check, consistent with this amendment being scoped to documentation only, not new automation.

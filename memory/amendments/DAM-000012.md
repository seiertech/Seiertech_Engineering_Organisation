## 3. Scope Decision — Why These 10, Not All 25

Stated explicitly, not left implicit: these 10 role-areas were chosen because they either gate real decisions (Security Architect's CRITICAL veto, Chief Architect's TDA verdict, Standards Engineer's conformance gate, Verification's PASS/FAIL, Release Manager's RELEASE/HOLD/REJECT) or directly drive the content quality the BUILD chain depends on (Data Architect, the engineering leads, AI Architect, Technical Debt Auditor). The remaining 15 role-areas were not upgraded in this amendment — this is a deliberate priority choice, named here rather than silently partial, consistent with the same disclosure discipline applied to `DAM-000010`'s intake gap closure.

---

## 4. Before / After — The Actual Shape of the Change

**Before, representative example (Security Architect):** "Always: Run dependency scan and flag CVEs above MEDIUM severity." Procedurally correct, but doesn't distinguish a CVSS 9.8 with no known exploit from a CVSS 7.5 under active exploitation, doesn't address direct-vs-transitive dependency risk, doesn't name any specific authentication anti-pattern to look for.

**After:** Each upgraded persona's AI Reasoning Profile now includes a dedicated section naming specific, checkable patterns — e.g. Security Architect now specifies JWT algorithm-confusion checks, IDOR detection via missing data-layer authorisation, exploit-maturity-aware CVE triage; Chief Architect now names circular dependencies, leaky abstractions, and pattern duplication as specific architectural smells with severity guidance distinguishing structural defects from style preference; Verification and Governance Director now distinguishes a test that genuinely exercises an acceptance criterion's condition from one that merely shares a name with the relevant area.

Every upgrade follows the same shape: replace "do X" with "here is specifically what X looks like, what doesn't count as X, and how to tell the difference" — the thing a genuine domain expert would actually bring that a generic instruction does not.

---

## 5. Why This Amendment Resolves The Lesson

LES-000020's finding was that content was generic rather than expert. The resolution is a direct, file-by-file content rewrite adding the missing specificity — not a structural change, not a new section, but the actual reasoning instructions themselves upgraded. This is verifiable by direct comparison: every upgraded section is longer, more specific, and contains named patterns/criteria that a templated generic role description would not include.

## 7. Regression Prevention

Content-depth quality is not mechanically checkable the way structural completeness is — there is no automated test for "is this reasoning instruction genuinely expert-level." The closest available substitute, used here, is direct comparison against what a real domain practitioner would need to know, performed by reading and rewriting each profile individually rather than templating a fix across all of them. This is acknowledged as a real limitation: the remaining 15 unupgraded role-areas have not been checked against this same bar, and even the 10 upgraded here have not been validated against a live model's actual output quality, since no live NIM call has been made this session.

## 3. The Terminal State Model

| Label | Condition |
|---|---|
| `status:ambiguous-mission` | Parse failed — now applied, was previously comment-only |
| `status:blocked` | Chain ran, platform genuinely not ready (a gate beyond the known RG-008 ceiling failed) OR a BUILD run produced an unrecognised/UNKNOWN status (fallback, never silently defaults to a review-looking label) |
| `status:draft-needs-review` | Chain hit the honest current ceiling — only RG-008 failing, the expected best outcome until Founder Questions exists |
| `status:halted-at-tda` | BUILD stopped because TDA returned REJECTED or REVISION_REQUIRED |
| `status:release-hold` | BUILD reached Release, decision was HOLD |
| `status:release-rejected` | BUILD reached Release, decision was REJECT |
| `status:released-not-delivered` | BUILD reached RELEASE but cross-repo delivery failed, was skipped, or `TARGET_REPO_TOKEN` is not configured |
| `status:delivered-to-target` | BUILD reached RELEASE and a real PR now exists in the target platform repo |

**Deliberately excluded: `status:complete`.** Per `EMS_OPERATING_MODEL.md` Section 5, a mission is only formally COMPLETE once `OPR-000008` Knowledge Capture has executed — that operation has no executor yet. Applying `status:complete` here would overclaim a closure the workflow cannot actually back up. This is stated explicitly in the workflow's own header comment, not left as a silent omission.

## 4. Point-by-Point Resolution of the Five Findings

1. **Ambiguous missions, no label** — fixed. The ambiguous-mission step now applies `status:ambiguous-mission` before posting its comment.
2. **All successful chains end in the same flat label** — fixed for all three chains. Intake and Genesis now read the actual gate-check failing-gates list and select `status:draft-needs-review` only when the failure is exclusively the known RG-008 ceiling; any other failure gets `status:blocked`. BUILD reads the actual `Final Status` value from the run log and selects from 5 distinct labels based on it.
3. **Released-but-undelivered indistinguishable from delivered** — fixed. The delivery step's `id: deliver` output (`steps.deliver.outcome`, GitHub Actions' built-in per-step success/failure signal) is read directly to distinguish `status:delivered-to-target` from `status:released-not-delivered`. The issue comment was also updated to state this distinction in prose, not just the label.
4. **TDA rejection halts correctly but gets the generic label** — fixed. `Final Status` values beginning with `HALTED_AT_TDA` (the exact string `run_build_chain.py` already writes) now map to `status:halted-at-tda` specifically.
5. **Lesson-worthy findings surfaced but not filed** — acknowledged by the reviewer as an acceptable current hand-off gap, not fixed in this amendment. Recorded here as a known remaining item, not silently dropped.

---

## 5. Why This Amendment Resolves The Lesson

Each of the 5 findings maps to a specific, traceable code change, not a general tidy-up. The label selection logic now reads the actual data already being collected (gate check results, BUILD final status, delivery step outcome) rather than ignoring it in favour of a hardcoded default — the data needed to make the right decision was already present in every case, it just wasn't being used.

## 7. Regression Prevention

The GitHub Actions JavaScript label-selection logic itself is not unit-testable in this environment — same disclosed limitation as `DAM-000005` and `DAM-000006` for other GitHub Actions wiring. The Python port-and-simulate approach used here for verification is the closest available substitute and is now the established pattern for this class of risk in this repo (first used for this amendment, worth reusing for any future workflow label/branching logic).

# LES-000008 — No single authoritative operating model document exists

| Field | Value |
|---|---|
| Lesson ID | LES-000008 |

| Source | External review — "Autonomous Engineering Management System (EMS) Findings & Recommendations Review," 30 June 2026 |
| Date | 2026-06-30 |
| Doctrine Amendment | DAM-000004 |
| Status | AMENDED |

## Finding
The review identified, correctly, that "how engineering work enters the system, how autonomous work progresses, how engineering decisions are made, how authority is applied, how engineering work completes" has no single authoritative description. The EMS defines this implicitly across 12 separate Operations documents (OPR-000001 through OPR-000012), each describing one slice of process, but a reader has to assemble the full picture by reading all of them and inferring the connections. The review's own framing is precise: the repository defines how engineering artefacts are governed, not yet how autonomous engineering itself operates.

## Impact
A new persona instance, a new contributor, or the founder returning after time away has no single entry point to understand the full operating model. This also makes it harder to spot gaps in the model itself — the kind of structural bug this session's simulation exercise found (the greenfield gate-checker bug) was found by exercising one path through the system, not by reading a complete model and reasoning about all paths.

## Reuse Guidance
Before adding the 13th Operation or further expanding any single slice of process, produce and maintain one authoritative Operating Model document that synthesises the existing Operations rather than replacing them — same pattern as a Master Technical Specification synthesising platform-level artefacts. Treat operating-model coherence as a thing to actively check, not something that emerges automatically from having enough Operations documents.

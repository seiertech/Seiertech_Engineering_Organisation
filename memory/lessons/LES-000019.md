# LES-000019 — A specific str_replace pattern repeatedly dropped a document header

| Field | Value |
|---|---|
| Lesson ID | LES-000019 |
| Category | STRUCTURAL_BUG |
| Source | Self-observed across this session — noticed on the fourth occurrence |
| Date | 2026-06-30 |
| Doctrine Amendment | DAM-000011 |
| Status | AMENDED |

## Finding
When inserting a new section immediately before an existing `## ` header by using `str_replace` with `old_str` set to only the header line itself, the header was silently dropped 4 separate times across this session — always in `IMPLEMENTATION_STATUS.md`, always caught and fixed only by a follow-up `grep` check, never caught before the edit was made. Root cause: this pattern requires the `new_str` to end by re-including the header verbatim (new content + the original header), and on at least 4 occasions the header was left out of `new_str`, silently deleting it rather than prepending before it.

## Impact
Low severity per-occurrence (caught every time before commit, never shipped broken), but the repetition itself is the real finding — the same specific mistake recurring 4 times in one session against the same file indicates a pattern in how that particular edit shape is being executed, not 4 independent slips.

## Reuse Guidance
When inserting a new section directly before an existing header, prefer anchoring `old_str` to a short, unique snippet of content immediately *after* the header (e.g. the first line of the table or paragraph that follows it) rather than the header text itself, and include the header in `new_str` only as context, never as the sole content being matched and replaced. Alternatively, always immediately `grep` for the header text right after this specific edit shape — not just at the end of a batch of edits — since this is the one edit pattern in this session that has reliably produced a silent drop.

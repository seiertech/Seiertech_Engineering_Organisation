# LES-000011 — No mechanism persisted the target repo URL for forward missions

| Field | Value |
|---|---|
| Lesson ID | LES-000011 |
| Category | DOCTRINE_GAP |
| Source | Solving the "cross-repo write capability" fundamental |
| Date | 2026-06-30 |
| Doctrine Amendment | DAM-000006 |
| Status | AMENDED |

## Finding
`MISSION-001` Platform Intake is handed the target repo URL directly as a parameter at the moment it runs (`--repo-url`), and uses it to clone and scan the repo. Nothing in `run_intake_chain.py`, the platform directory structure, or any doctrine artefact ever wrote that URL down anywhere a *later* mission could read it. `OPR-000005` Engineering Delivery Operation describes creating a GitHub branch and Pull Request as part of delivery, but doctrine never specified where the target lives for that step to act against. Found while designing `deliver_to_target_repo.py` and realising there was no existing source for the URL it needed.

## Impact
Without this fix, every BUILD (or future REHAB/etc) mission for a platform would have had no mechanism to know which repository to deliver to, regardless of how complete the rest of the BUILD chain was — a second instance of the same class of gap as LES-000002 (greenfield gate-checker) and LES-000010 (genesis brief mapping): a downstream capability silently depending on upstream data that was never actually persisted.

## Reuse Guidance
When a script or operation is handed a piece of data at one point in the mission lifecycle that a later, separate mission will need, explicitly check whether that data is persisted anywhere durable before assuming it will be available later. The fix here — a single-line marker file (`PLATFORM_REPO_URL.txt`) written once during intake — is intentionally minimal; the general pattern worth repeating is: identify what later missions will need, write it down once at the earliest point it's known, in a location that's easy for a human to inspect or correct by hand.

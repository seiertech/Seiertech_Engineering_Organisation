# LES-000021 — Workflow had no real terminal state model

| Field | Value |
|---|---|
| Lesson ID | LES-000021 |
| Category | STRUCTURAL_BUG |
| Source | External review of the mission chain workflow |
| Date | 2026-06-30 |
| Doctrine Amendment | DAM-000013 |
| Status | AMENDED |

## Finding

An external reviewer examined `ems-mission-chain.yml` directly and found five real, confirmed-accurate dead-end risks:

1. **Ambiguous missions got a comment but no label.** A parse failure posted an explanatory comment with no status label, meaning the issue could silently sit unactioned in the tracker with no way to filter for it.
2. **Every successful chain outcome collapsed into one flat label.** Intake, Genesis, and BUILD all removed `status:in-progress` and unconditionally applied `status:draft-needs-review`, regardless of whether the actual outcome was READY, BLOCKED, a TDA halt, a HOLD, a REJECT, a successful delivery, or a failed delivery.
3. **A RELEASE with failed cross-repo delivery looked identical to a fully delivered mission.** The delivery step is `continue-on-error: true` by design (so a missing `TARGET_REPO_TOKEN` doesn't block the rest of the BUILD chain), but nothing distinguished "released and a real PR exists" from "released but nothing was pushed anywhere" in the resulting label.
4. **TDA rejection halted correctly but still got the generic review label.** A REJECTED or REVISION_REQUIRED TDA verdict is the chain working as designed — but the issue ended up with the same `status:draft-needs-review` label as a genuine success, hiding that a real halt occurred.
5. **Lesson-worthy findings surfaced in BUILD output were never auto-filed.** Acknowledged by the reviewer as an acceptable known hand-off gap, not a structural defect requiring the same urgency as 1-4.

## Impact

A human or automated process scanning issues by label could not distinguish a genuinely successful, ready-for-review outcome from a genuinely blocked, rejected, or undelivered one — every meaningfully different ending looked the same from the outside, defeating the purpose of having status labels at all.

## Reuse Guidance

When a workflow has multiple distinct possible outcomes, the terminal label/state must be selected based on the actual outcome of the run, not applied as a single hardcoded default at the end of every path. This is the same class of finding as conflating "the chain runs without error" with "the chain produced a correct/complete result" — surfaced earlier this session for intake depth (LES-000018) and now again for workflow terminal states. The pattern is worth naming generally: success of execution and quality/completeness of outcome are different claims, and a workflow's status reporting should reflect the latter, not just the former.

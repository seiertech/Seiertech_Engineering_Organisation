# LES-000023 — Gates step skip on chain crash causes incorrect status:draft-needs-review

| Field | Value |
|---|---|
| Lesson ID | LES-000023 |
| Category | STRUCTURAL_BUG |
| Source | Right-to-left workflow logic sweep |
| Date | 2026-06-30 |
| Doctrine Amendment | DAM-000014 |
| Status | AMENDED |

## Finding
In intake and genesis jobs, the "Read gate check result" step had no `if:` condition, meaning it only runs if all preceding steps succeed (GitHub Actions' default). The "Update status label" step had `if: always()`. If the chain itself failed (e.g. no NIM_API_KEY, network error) the gates step would be skipped, `FAILING_GATES` would be empty or unset, and the label logic would interpret that as "no failures" → `status:draft-needs-review`. A crash was producing the same label as a successful run at the expected ceiling.

## Impact
A script crash or configuration failure would silently look like a normal review-awaiting result from the issue tracker, hiding that anything went wrong. The only indication would be the failed GitHub Actions run in the repo, not the issue label visible to the mission operator.

## Reuse Guidance
Any step that reads state to inform a downstream `if: always()` step must itself have `if: always()` — otherwise the reader is skipped on failure and the downstream step reads stale/empty state. The general pattern: if a terminal status step runs unconditionally, everything that feeds it should also run unconditionally, with appropriate fallback values for the failure case.

## Finding
`run_intake_chain.py` (v2) produces 6 artefacts: `USE_CASE_REGISTER.md`, `ARCHITECTURE_DOCUMENT.md`, `SECURITY_POSTURE.md`, `TECHNICAL_DEBT_REGISTER.md`, `KNOWLEDGE_GRAPH.md`, `MASTER_TECHNICAL_SPECIFICATION.md`. `MSN-000001` doctrine specifies 17 Layer 1 artefacts plus 3 Layer 3 synthesis artefacts plus the MTS — 21 distinct named artefacts beyond what's actually produced.

Cross-referencing every Team 2 persona's stated `Inputs` against the real 6-artefact output found 16 of 25 personas (64%) reference at least one input that does not exist: Data Architect (Data Model), every engineering lead (Frontend/Backend/Integration/Platform Assessments, API Register), the AI Architect (AI Capability Map), Verification (Test Strategy), Enterprise Architect (Enterprise Architecture Context), Knowledge Graph Architect (Domain Vocabulary), Documentation Curator (Documentation Assessment), both Proposition-related personas, and critically both top-level governance personas (Executive Director, Mission Control Director — both expect a Handoff Artefact that is never produced by the real chain).

The MTS synthesis pass itself draws from only 5 of the doctrine's intended ~17+ source documents. Cross-checked against the doctrine-specified 15 MTS sections: real content exists for roughly 6, nothing exists for 9 (API Specification, Integration Specification, Frontend Specification, Backend Specification, Deployment Specification, AI Capability Specification, Test Specification, Proposition Summary, and a genuine Mission Readiness Declaration, since the gate check runs after MTS synthesis not before).

`run_build_chain.py` itself only reads `MASTER_TECHNICAL_SPECIFICATION.md` and `ARCHITECTURE_DOCUMENT.md` — both of which exist — which is why BUILD missions execute without crashing. This masked the severity of the gap: the chain *runs*, but Team 2 is reasoning with roughly a quarter of the context its own persona definitions assume it has.

## Impact
Every BUILD mission run to date (test fixtures only, no live calls yet) has been reasoning against an MTS missing API surface, deployment model, frontend/backend specifics, AI capability mapping, and test coverage information — exactly the kind of detail that distinguishes a sound TDA verdict from a guess.

## Reuse Guidance
"The chain runs without error" is not the same claim as "the chain has sufficient context" — these must be checked separately. The correct verification method, used here, is cross-referencing each consuming persona's stated Inputs against the actual producing script's real output, not just confirming the script executes cleanly against a fixture.

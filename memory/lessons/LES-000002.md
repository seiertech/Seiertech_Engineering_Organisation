# LES-000002 — Gate checker structurally could never pass a greenfield platform

| Field | Value |
|---|---|
| Lesson ID | LES-000002 |
| Category | STRUCTURAL_BUG |
| Source | Greenfield simulation exercise |
| Date | 2026-06-29 |
| Doctrine Amendment | DAM-000001 |
| Status | AMENDED |

## Finding
`check_readiness_gates.py`'s RG-001/002/004 and the Standards Engineer gate only recognised brownfield-only evidence files (`SCAN_RESULT.json`, `INTAKE_RUN_LOG.md`). Genesis platforms have no repo to scan by design — meaning a greenfield platform could never produce these files and was therefore structurally blocked from ever reaching READY, regardless of how complete its genesis run was. Found by building a deliberately complete simulated greenfield artefact set and watching it still fail.

## Impact
This was the most severe of the three simulation bugs — it would have made MISSION-000 Genesis entirely unusable end-to-end even once an executor existed, silently, with no clear error pointing at the actual cause.

## Reuse Guidance
When a shared mechanism (here, the gate checker) is built while only one of two intended use cases (brownfield) exists, explicitly test it against the second use case (greenfield) before assuming it generalises — "it works for the case I built it against" is not the same as "it works."

# LES-000015 — Commander onboarding checklist never swept into platform-agnostic generalisation

| Field | Value |
|---|---|
| Lesson ID | LES-000015 |
| Category | DOCTRINE_GAP |
| Source | Direct founder challenge ("why the fuck is this still Commander onboarding") |
| Date | 2026-06-30 |
| Doctrine Amendment | DAM-000008 |
| Status | AMENDED |

## Finding
`COMMANDER_ONBOARDING_CHECKLIST.md` remained Commander-specific by filename and content throughout this entire session, despite a dedicated effort earlier the same session (`PLATFORM_AGNOSTIC_AUDIT.md`) finding and fixing 22 instances of exactly this class of problem across Team 2 personas. That audit checked `agents/`, `missions/`, `standards/`, `operations/` — it never checked root-level documents, so this one was never in scope to be caught by it.

## Impact
A document genuinely useful and correct in content was undermined by its own framing — every reference to it implicitly suggested it only applied to Commander, the opposite of the EMS's stated platform-agnostic design principle, for the entire session until directly challenged.

## Reuse Guidance
When running a platform-agnostic audit, explicitly include root-level documents in scope, not just the structured doctrine folders (`agents/`, `missions/`, `standards/`, etc). Root-level documents are exactly where session-specific working notes are most likely to be written first and then never swept back into the general pattern, because they don't live inside a folder whose name implies "this should be generic."

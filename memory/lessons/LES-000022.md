# LES-000022 — issue_comment trigger causes re-fire loop

| Field | Value |
|---|---|
| Lesson ID | LES-000022 |

| Date | 2026-06-30 |

| Status | AMENDED |

## Finding
The workflow had `issue_comment: [created]` as a trigger, paired with `${{ github.event.issue.title }}` as the input to the issue parser. This means ANY comment on any mission issue — including the EMS bot's own status comments — would re-fire the detect-mission job, re-parse the original issue title, and route to a second (third, infinite...) execution of the same chain. Confirmed by testing: posting "This is a comment reply" to an existing INTAKE issue re-triggers the full intake chain.

## Impact
On the first real run against a real platform, the intake chain would complete, post a status comment, that comment would trigger another intake run, which would post another comment, which would trigger another intake run. An actual infinite loop at production cost.

## Reuse Guidance
The `issue_comment` trigger has a different event payload structure from the `issues` trigger — `github.event.issue` exists in both, but adding `issue_comment` to the trigger list means every comment becomes a new trigger. Always verify a new trigger type's actual effect against the existing parsing logic before adding it. If re-parsing on comment is genuinely wanted, add a condition gating on the commenter not being a bot actor: `github.actor != 'github-actions[bot]'`.

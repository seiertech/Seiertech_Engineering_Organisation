## 1. Purpose

This standard absorbs the useful part of loop engineering into EMS without changing the EMS operating structure.

It defines the minimum contract for any bounded EMS execution loop so that missions, workflow jobs, Python executors, AI persona passes and review gates are triggered, governed, verified, stopped and converted into artefacts consistently.

This standard does not replace Mission Orders, Operations, Authorities, Registers, persona doctrine or the EMS Operating Model. It is a lower-level execution grammar that those artefacts may use.

---

## 2. Scope

This standard applies to:

- Mission chain workflows.
- Intake, Genesis and BUILD executors.
- Future REHAB, STRATEGIC, AGENTIC_INSERTION, SPEC and PROPOSAL executors.
- AI persona passes.
- Standards Engineer gates.
- TDA, Verification, Release and Knowledge Capture stages.
- Any future autonomous engineering loop added to EMS.

---

## 3. EMS Loop Contract

Every bounded EMS execution loop SHALL be describable by the following contract:

| Contract Field | Requirement |
|---|---|
| Trigger | The event, issue, comment, schedule, gate result or upstream artefact that starts the loop. |
| Goal | The bounded outcome the loop is attempting to produce. |
| Work Unit | The smallest meaningful unit of work executed by the loop. |
| Actor | The persona, script, workflow job or human authority performing the work unit. |
| Verifier | The independent persona, script, gate or authority that validates the work. |
| Governor | The authority, standard, budget, scope, permission rule or approval gate limiting the loop. |
| Stop Condition | The explicit condition that ends the loop, whether successful or not. |
| Output Artefact | The file, register entry, issue comment, PR, memory entry or decision record produced by the loop. |
| Knowledge Update | Any lesson, memory entry, doctrine amendment trigger or register update produced by the loop. |

A loop may be implemented in Markdown doctrine, GitHub Actions, Python, human procedure or a combination of these, but the contract SHALL remain recognisable.

---

## 4. Standard Stop Conditions

EMS loops SHALL use explicit stop conditions rather than vague completion language.

Permitted standard stop conditions are:

| Stop Condition | Meaning |
|---|---|
| COMPLETE | The bounded loop achieved its stated goal and produced the required artefact. |
| READY_WITH_KNOWN_CEILING | The loop achieved the best currently honest state, but a known system-wide gap remains. |
| BLOCKED_BY_GATE | A deterministic gate prevents continuation. |
| HALTED_BY_AUTHORITY | A governing authority, such as TDA or Security veto, stopped the loop. |
| VERIFIER_FAILED | Independent verification failed. |
| RELEASE_HOLD | Release was not rejected but must not proceed yet. |
| RELEASE_REJECTED | Release was rejected and the reasoning must be preserved. |
| DELIVERED_TO_TARGET | The loop delivered its output to the target repository or downstream system. |
| DELIVERY_FAILED | Release succeeded locally but downstream delivery failed or lacked credentials. |
| DEPENDENCY_MISSING | A required system, token, repo, artefact or integration is absent. |
| NO_EVIDENCE | The loop cannot continue because the required evidence is absent. |
| NO_PROGRESS | The loop repeated or attempted work without producing improved evidence or artefacts. |
| HUMAN_REQUIRED | The loop requires founder, authority or human review before continuing. |
| AMBIGUOUS_INPUT | The trigger could not be parsed into a known mission type or loop pattern. |
| TIME_OR_BUDGET_EXHAUSTED | The allowed runtime, token budget or execution budget was exceeded. |

Any new stop condition SHALL be added to this standard before it becomes routine workflow behaviour.

---

## 5. Actor / Verifier Separation

The actor that produces a work unit SHALL NOT be the final verifier of that same work unit.

Acceptable verifier examples include:

- Standards Engineer gate checking an artefact generated by another persona.
- TDA reviewing an Engineering Proposal generated by product or analyst personas.
- Verification and Governance Director reviewing an Engineering Delivery Package generated by engineering personas.
- Deterministic Python checks validating workflow output.
- Release Manager applying the Release Scorecard after Verification.

Self-assessment may be used as supporting context, but it SHALL NOT be treated as final verification.

---

## 6. Governor Definition

For EMS purposes, a Governor is the set of constraints that bounds a loop.

A Governor may include:

- Constitutional authority.
- Mission authority.
- AI governance requirements.
- Knowledge governance requirements.
- Role authority.
- Standards conformance.
- Deterministic gates.
- Approval or veto rules.
- Repository permissions.
- Token, runtime or budget ceilings.
- Scope limits and known exclusions.

Every executable EMS loop SHOULD cite at least one Governor.

## 7. Minimum Loop Pattern Library

The following loop patterns are recognised as EMS baseline patterns:

| Pattern | Description |
|---|---|
| Intake Loop | Brownfield platform scan, persona passes, MTS synthesis, handoff and readiness gating. |
| Genesis Loop | Greenfield platform design from brief, MTS synthesis and readiness gating. |
| BUILD Loop | Proposal, TDA, EDP, Verification, Release and Knowledge Capture. |
| TDA Review Loop | Proposal review against architecture and authority. |
| Verification Loop | Independent acceptance-criteria validation of produced delivery artefacts. |
| Release Loop | Scorecard-based RELEASE/HOLD/REJECT decision. |
| Knowledge Capture Loop | Memory update, lesson detection and doctrine amendment trigger. |
| Doctrine Drift Loop | Detection and correction of drift between doctrine, scripts, workflows and registers. |
| Persona Improvement Loop | Capture of missing persona inputs, weak reasoning or depth gaps into doctrine amendments. |
| Workflow Repair Loop | Deterministic correction of workflow failure modes and terminal-state ambiguity. |

---

## 8. Conformance Requirements

An EMS loop is conformant when:

1. Its trigger is explicit.
2. Its goal is bounded.
3. Its actor and verifier are distinguishable.
4. Its governor is cited or inferable from doctrine.
5. Its stop condition is explicit.
6. Its output artefact is named.
7. Its knowledge update behaviour is stated.
8. It avoids claiming COMPLETE until Knowledge Capture has executed where mission completion is in scope.

---

## 9. Left-to-Right / Right-to-Left Sweep Requirement

When this standard is introduced or materially amended, EMS SHALL perform a two-direction sweep:

- Left-to-right: Constitution → Authorities → Standards → Registers → Operations → Templates → Personas → Workflows → Python scripts → Memory.
- Right-to-left: Python scripts → Workflows → Personas → Templates → Operations → Registers → Standards → Authorities → Constitution.

The sweep must confirm that the loop contract is not merely documented, but wired into execution-relevant artefacts.
